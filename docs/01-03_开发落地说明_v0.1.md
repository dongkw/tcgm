# 01-03 开发落地说明 v0.1

> 本文档把 01 数据目录、02 策略引擎、03 时间模块翻译成第一轮可开发的最小闭环。  
> 当前仍然只写设计，不写代码。  
> 第一轮目标不是做完整系统，而是验证核心链路是否跑得通。

---

## 1. 第一轮开发目标

第一轮只做一个最小闭环：

```text
现有 stock_data_<code>.json
  ↓
time_context
  ↓
strategy_snapshot
  ↓
strategy_engine
  ↓
decision_result
  ↓
JSON / Markdown 输出
```

第一轮不做：

- SQLite。
- Web 界面。
- 插件输出。
- 自动交易。
- 模拟盘成交。
- 全市场扫描。
- 新数据源重构。

第一轮要验证：

- 现有数据能否转换成策略快照。
- 策略引擎能否只读 `strategy_snapshot`。
- 时间校验能否阻止未来数据。
- 数据缺失能否降级。
- 持仓卖出能否在缺成本价时标记预评估。
- 输出是否能被人读懂，也能被机器继续处理。

## 2. 第一轮输入输出

### 2.1 输入

第一轮输入来自现有脚本生成的文件：

```text
data/stock_json/stock_data_<股票代码>.json
```

例如：

```text
data/stock_json/stock_data_002563.json
data/stock_json/stock_data_002653.json
```

该文件在第一轮中视为：

```text
raw_source
```

暂时不要求重新拆出完整 `raw_data` 表。

### 2.2 输出

第一轮输出两个 JSON：

```text
data/strategy_snapshots/strategy_snapshot_<代码>_<时间>.json
data/decision_results/decision_result_<代码>_<时间>.json
```

可选输出一个 Markdown：

```text
data/reports/decision_report_<代码>_<时间>.md
```

### 2.3 第一轮目录

建议新增目录：

```text
data/
├── strategy_snapshots/
├── decision_results/
└── reports/
```

暂不新增数据库目录。

## 3. 第一轮处理流程

### 3.1 总流程

```text
load_stock_json(code)
  ↓
build_time_context(raw_json, task_type)
  ↓
normalize_stock_data(raw_json)
  ↓
quality_check(normalized_data, time_context, task_type)
  ↓
build_strategy_snapshot(normalized_data, time_context, quality_result, user_context)
  ↓
run_strategy(strategy_snapshot)
  ↓
write_decision_result(decision_result)
```

### 3.2 模式

第一轮支持两种模式：

```text
BUY_EVALUATION   买入评估
HOLDING_REVIEW   持仓卖出/持有评估
```

`WATCHLIST_SCAN` 可以后移。

### 3.3 用户补充上下文

对于持仓评估，用户可能需要补：

```text
avg_cost
position_pct
total_quantity
available_quantity
buy_logic
invalidation_point
holding_period
```

如果缺少这些，系统仍可生成 `strategy_snapshot`，但 `decision_result.final_action` 最高只能是：

```text
PRE_EVALUATION
```

## 4. time_context 契约

每次生成策略快照前，先生成 `time_context`。

### 4.1 字段

```json
{
  "timezone": "Asia/Shanghai",
  "calendar_date": "2026-07-04",
  "decision_time": "2026-07-04T10:00:00+08:00",
  "task_type": "HOLDING_REVIEW",
  "exchange": "SZ",
  "trade_date": "2026-07-03",
  "is_trading_day": false,
  "session_name": "NON_TRADING",
  "previous_trade_date": "2026-07-03",
  "next_trade_date": null,
  "effective_data_cutoff": "2026-07-04T10:00:00+08:00",
  "allowed_data_types": [
    "DAILY_CLOSE",
    "FINANCIAL_REPORT",
    "ANNOUNCEMENT",
    "POSITION_SNAPSHOT"
  ],
  "blocked_data_reasons": [],
  "time_warnings": [
    "next_trade_date 未接入正式交易日历，第一轮仅做保守校验"
  ]
}
```

### 4.2 第一轮简化规则

第一轮没有完整交易日历时：

- `trade_date` 直接使用原始 JSON 的 `quote.trade_date`。
- `decision_time` 使用系统当前时间。
- 非交易时段也允许生成分析报告。
- 任何涉及真实交易执行的动作都标记为 `human_review_required = true`。
- T+1 只校验已有 `available_quantity`，不自动推断交易日。

### 4.3 必须保留的约束

即使第一轮简化，也必须保留：

- 数据不能晚于 `decision_time`。
- 财务数据必须有 `notice_date`。
- 公告数据必须有日期或链接，否则只能做提示。
- 持仓卖出必须尊重 `available_quantity`。

## 8. 数据质量与降级规则

### 8.1 买入评估

| 缺失或异常 | 处理 |
|---|---|
| 当前价缺失 | `DATA_BLOCKED` |
| 交易日缺失 | `DATA_BLOCKED` |
| 20/60 日线缺失 | 最高 `WAIT` |
| 最新财务核心字段缺失 | `DATA_BLOCKED` |
| 财报 `notice_date` 缺失 | 财务规则不可用于硬判断，最高 `WAIT` |
| 估值分位缺失 | 估值规则降级，最高 `WATCH_SMALL` |
| 公告原文未复核 | 最高 `WATCH_SMALL` |
| 现金数据缺失 | 不能给具体买入金额和数量 |

### 8.2 持仓评估

| 缺失或异常 | 处理 |
|---|---|
| 当前价缺失 | `DATA_BLOCKED` |
| 20/60 日线缺失 | 技术卖出规则降级 |
| 成本价缺失 | `PRE_EVALUATION` |
| 当前仓位缺失 | `PRE_EVALUATION` |
| 原买入逻辑缺失 | `PRE_EVALUATION` |
| 原证伪点缺失 | `PRE_EVALUATION` |
| 可卖数量缺失 | 不能给可执行卖出数量 |
| 可卖数量为 0 且触发卖出 | `NO_SELL_T_PLUS` |

## 9. 第一轮策略规则范围

### 9.1 买入 MVP

第一轮买入只判断：

- 数据是否足够。
- 是否命中前置硬否决。
- 基本面是否有明显硬伤。
- 是否追高。
- 是否站上 20/60 日线。
- 是否有买点或只能等待。
- 是否能给观察仓。

第一轮买入不做：

- 复杂 EPS 预测。
- 行业深度判断。
- 多因子打分。
- 自动给大仓位。

### 9.2 卖出 MVP

第一轮卖出判断：

- 缺持仓信息时给 `PRE_EVALUATION`。
- 跌破 20 日线触发短线减仓。
- 跌破近 20 日低点触发清仓或继续减仓。
- 跌破 60 日线触发中线减仓。
- 1 个月涨幅过大触发止盈提示。
- 无可卖数量时标记 T+1 限制。

## 12. 验收样例

### 12.1 样例 1：002563 持仓预评估

输入：

```text
data/stock_json/stock_data_002563.json
task_type = HOLDING_REVIEW
不提供成本价、仓位、买入逻辑
```

预期：

- 能生成 `strategy_snapshot`。
- `position.provided = false`。
- `decision_result.final_action = PRE_EVALUATION`。
- 报告中提示缺持仓信息。
- 技术价位仍能输出：20 日线、60 日线、近 20 日高低点。

### 12.2 样例 2：002653 持仓预评估

输入：

```text
data/stock_json/stock_data_002653.json
task_type = HOLDING_REVIEW
不提供成本价、仓位、买入逻辑
```

预期：

- 能生成 `strategy_snapshot`。
- 能识别近 20 日涨幅较大。
- 能输出反弹压力、20 日线、60 日线、近 20 日低点。
- 因缺持仓信息，最终仍是 `PRE_EVALUATION`。

### 12.3 样例 3：缺财务字段

输入：

```text
删除 financial.latest.deducted_net_profit_yuan
task_type = BUY_EVALUATION
```

预期：

- `data_quality.blocking_missing_fields` 包含该字段。
- `decision_result.final_action = DATA_BLOCKED`。
- 不允许输出 `BUY`。

### 12.4 样例 4：策略不能读 raw

验收：

- 策略函数输入只有 `strategy_snapshot`。
- 策略函数不接收原始 `stock_data`。
- `decision_result.snapshot_id` 必须存在。

## 13. 第一轮开发顺序

建议按下面顺序开发：

1. 读取现有 `stock_data_<code>.json`。
2. 生成 `time_context`。
3. 生成 `strategy_snapshot`。
4. 生成 `data_quality`。
5. 实现最小卖出预评估。
6. 实现最小买入评估。
7. 输出 `decision_result`。
8. 输出简单 Markdown 报告。
9. 用 002563 和 002653 验证。
10. 根据验证结果修正文档。

第一轮优先做卖出/持仓评估，因为它更贴近当前使用场景。

---

## 14. 第一轮完成标准

第一轮代码完成后，应满足：

- `002563` 能从现有 JSON 生成 `strategy_snapshot`。
- `002653` 能从现有 JSON 生成 `strategy_snapshot`。
- 策略引擎只读取 `strategy_snapshot`。
- 每次决策都生成 `decision_result`。
- `decision_result` 引用 `snapshot_id`。
- 缺持仓信息时输出 `PRE_EVALUATION`。
- 缺关键财务字段时输出 `DATA_BLOCKED`。
- 报告能列出关键价位和数据缺口。
- 不写入数据库也能跑通最小闭环。
