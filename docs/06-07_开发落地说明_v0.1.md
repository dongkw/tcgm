# 06-07 开发落地说明 v0.1

> 本文档把 06 组合构建、07 风控分层翻译成第三轮可开发的最小闭环。  
> 当前仍然只写设计，不写代码。  
> 第三轮目标不是做复杂组合优化，而是让多个 `decision_result` 能经过硬风控、排序和资金分配，生成可确认的买入计划。

---

## 1. 第三轮开发目标

第三轮只做一个最小闭环：

```text
多个 decision_result
  ↓
candidate_decision
  ↓
risk_check_result
  ↓
ranked_candidate
  ↓
allocation_plan
  ↓
order_intent
  ↓
allocation_report
```

第三轮要验证：

- 多只股票同时触发时能排序。
- 不可交易结论不能进入买入计划。
- 非交易日、价格缺失、现金不足、超仓等硬风险能被拦截。
- 本金有限时能明确买谁、买多少、放弃谁。
- 买入计划能追溯到 `decision_id` 和 `snapshot_id`。
- 组合计划不直接成交，只输出给模拟盘或人工确认。

第三轮不做：

- 自动实盘下单。
- 复杂行业集中度。
- 协方差矩阵。
- 波动率优化。
- 机器学习选股排序。
- Redis / 任务队列。
- SQLite 正式迁移。
- Web 界面。

说明：

第三轮仍然沿用 JSON/JSONL 文件账本，等 08 数据库阶段再迁移 SQLite。

## 4. 开发模块建议

建议新增：

```text
src/ai_trader/risk_control.py
src/ai_trader/portfolio_construction.py
src/ai_trader/portfolio_plan.py
portfolio_plan.py
```

### 4.1 risk_control.py

职责：

- 校验数据和时间。
- 校验 action 是否可交易。
- 校验现金、单票仓位、T+1。
- 校验重复成交。
- 输出 `risk_check_result`。

不负责：

- 排序候选。
- 分配资金。
- 生成成交。
- 重新判断股票好坏。

### 4.2 portfolio_construction.py

职责：

- 汇总多个候选。
- 读取 `risk_check_result`。
- 计算透明评分。
- 按本金和仓位约束分配资金。
- 输出 `allocation_plan` 和 `order_intent`。

不负责：

- 风控硬拦截。
- 模拟成交。
- 调用 AI 排序。

### 4.3 portfolio_plan.py

职责：

- CLI 编排。
- 读取多个 `decision_result`。
- 回读对应 `strategy_snapshot`。
- 调用 `risk_control`。
- 调用 `portfolio_construction`。
- 写入 JSONL 和 Markdown 报告。

## 6. decision_result 读取规则

第一版只读取：

```text
task_type = BUY_EVALUATION
```

且只处理：

```text
final_action in BUY, WATCH_SMALL
```

其他 action 仍写入候选报告，但不参与资金分配：

```text
WAIT
DO_NOT_BUY
DATA_BLOCKED
HOLD
PRE_EVALUATION
NO_SELL_T_PLUS
```

读取候选时必须保留：

```text
decision_id
snapshot_id
symbol
name
task_type
decision_time
time_context
final_action
confidence
action_reason
position_plan
data_quality_summary
trigger_prices
```

## 7. strategy_snapshot 回读规则

第三轮必须回读与 `decision_result.snapshot_id` 对应的 `strategy_snapshot`。

用途：

- 获取 `quote.price`。
- 获取 `quote.trade_date`。
- 获取 `asset_type`。
- 获取估值、技术面、数据质量字段。

价格来源优先级：

```text
strategy_snapshot.quote.price
  ↓
decision_result.reference_price
  ↓
decision_result.quote.price
  ↓
data/stock_json/stock_data_<code>.json.quote.price
```

如果价格缺失：

```text
risk_status = BLOCK
reason = PRICE_MISSING
```

如果价格交易日与 `decision_result.time_context.trade_date` 不一致：

```text
risk_status = BLOCK
reason = PRICE_TIME_MISMATCH
```

## 8. 风控最小实现

### 8.1 风控入口

```text
check_decision_risk(decision, snapshot, account, positions, trade_history, config)
```

返回：

```text
risk_check_result
```

### 8.2 R0 数据和时间

阻塞条件：

```text
data_quality_summary.level = BLOCKED
price is missing
price <= 0
trade_date missing
price_trade_date != decision_trade_date
```

非交易日处理：

```text
time_context.is_trading_day = false
session_name = NON_TRADING
```

第一版组合计划允许在非交易日生成“计划”，但必须：

```text
risk_status = WARN
allowed_action = RECORD_ONLY_PLAN
execution_allowed = false
```

不能生成可成交 `order_intent`。

### 8.3 R1 action 校验

映射：

| final_action | 风控处理 |
|---|---|
| `BUY` | 可继续校验 |
| `WATCH_SMALL` | 可继续校验，小仓上限 |
| `WAIT` | `RECORD_ONLY` |
| `DO_NOT_BUY` | `REJECT` |
| `DATA_BLOCKED` | `BLOCK` |
| `HOLD` | `RECORD_ONLY` |
| `PRE_EVALUATION` | `RECORD_ONLY` |
| `NO_SELL_T_PLUS` | `RECORD_ONLY` |

### 8.4 R2 低信心

```text
confidence = LOW
```

处理：

```text
BUY -> WATCH_SMALL
WATCH_SMALL -> RECORD_ONLY
```

### 8.5 R3 现金和保留线

计算：

```text
cash_reserve = total_assets * cash_reserve_pct / 100
usable_cash = max(available_cash - cash_reserve, 0)
daily_budget_left = max_daily_buy_amount - today_buy_used
buy_budget = min(usable_cash, daily_budget_left)
```

如果 `buy_budget <= 0`：

```text
risk_status = REJECT
reason = NO_BUY_BUDGET
```

### 8.6 R4 单票仓位

买入后不得超过：

```text
total_assets * max_single_position_pct / 100
```

风控输出：

```text
max_cash_amount = min(buy_budget, single_position_remaining_value)
```

如果 `max_cash_amount` 不足 100 股：

```text
risk_status = REJECT
reason = CASH_NOT_ENOUGH_FOR_ONE_LOT
```

### 8.7 R5 防重复成交

从 `data/paper_trading/trades.jsonl` 读取历史成交。

如果已有相同 `decision_id` 成交：

```text
risk_status = BLOCK
reason = DUPLICATE_DECISION_EXECUTION
```

### 8.8 R6 daily_rollover

如果账户 `last_rollover_trade_date != trade_date`：

```text
risk_status = BLOCK
reason = DAILY_ROLLOVER_MISSING
```

例外：

非交易日研究模式只生成观察报告，不生成可执行 order_intent。

## 9. 组合评分最小算法

### 9.1 评分入口

```text
score_candidate(decision, snapshot, risk_check, portfolio_context, metadata)
```

### 9.2 分数结构

```text
total_score =
  signal_score
  + confidence_score
  + data_quality_score
  + portfolio_fit_score
  + user_priority_score
  - penalty_score
```

第一版不做：

```text
risk_reward_score
liquidity_score
industry_score
```

这些字段可以保留为 0，后续扩展。

### 9.3 signal_score

| final_action | 分数 |
|---|---:|
| `BUY` | 40 |
| `WATCH_SMALL` | 25 |

如果 `risk_check.allowed_action = WATCH_SMALL`：

```text
按 WATCH_SMALL 计分
```

### 9.4 confidence_score

| confidence | 分数 |
|---|---:|
| `HIGH` | 15 |
| `MEDIUM` | 8 |
| `LOW` | 0 |

### 9.5 data_quality_score

```text
data_quality_score = data_quality_summary.score * 0.2
```

缺失时：

```text
data_quality_score = 0
```

### 9.6 portfolio_fit_score

第一版：

```text
没有当前持仓 +5
已有当前持仓 0
已有当前持仓且仓位超过 20% -10
```

### 9.7 penalty_score

扣分：

```text
risk_status = WARN       5
risk_status = DOWNGRADE  10
confidence = LOW         10
data_quality_score < 75  10
```

`REJECT` 或 `BLOCK` 不进入排序。

## 10. 资金分配最小算法

### 10.1 候选过滤

只保留：

```text
risk_status in PASS, WARN, DOWNGRADE
allowed_action in BUY, WATCH_SMALL
```

### 10.2 排序

```text
按 total_score 降序
同分按 data_quality_score 降序
再同分按 decision_time 新到旧
```

### 10.3 默认目标金额

```text
BUY:
  target_cash = total_assets * 0.10

WATCH_SMALL:
  target_cash = total_assets * 0.05

DOWNGRADE:
  target_cash = min(target_cash, total_assets * 0.03)
```

如果 `decision_result.position_plan.suggested_cash_amount` 存在：

```text
target_cash = min(target_cash, suggested_cash_amount)
```

### 10.4 实际分配

```text
planned_cash = min(
  target_cash,
  risk_check.max_cash_amount,
  remaining_buy_budget
)
```

数量：

```text
raw_quantity = planned_cash / reference_price
planned_quantity = floor(raw_quantity / 100) * 100
```

如果 `planned_quantity < 100`：

```text
status = DEFERRED
reason = CASH_NOT_ENOUGH_FOR_ONE_LOT
```

实际占用金额：

```text
planned_cash_amount = planned_quantity * reference_price
```

第一版组合构建不计算手续费，最终现金校验仍由 `paper_trading` 再做一次。

### 10.5 资金递减

每生成一个 `READY_FOR_CONFIRM` 的 `order_intent`：

```text
remaining_buy_budget -= planned_cash_amount
```

剩余预算不足时，后续候选 `DEFERRED`。

## 11. allocation_report 中文说明

报告文件：

```text
data/reports/allocation_report_<account>_<trade_date>.md
```

内容：

```text
账户摘要
今日买入预算
候选总数
可买候选
被拒绝候选
延后候选
买入计划列表
每只股票评分拆解
每只股票资金分配原因
风控拦截原因
使用提示
```

必须明确：

- 这是计划，不是成交。
- 是否允许执行。
- 哪些股票因资金不足被放弃。
- 哪些股票因风控被拒绝。
- 哪些股票只是记录观察。

## 12. 与 paper_trading 的关系

第三轮先不强制改 `paper_trading.py`。

第一版关系：

```text
portfolio_plan.py 生成 order_intent 和报告
用户根据报告选择某个 decision_result
paper_trading.py apply ... --confirm 执行模拟成交
```

后续可扩展：

```text
paper_trading.py apply-intent <intent_id> --confirm
```

但必须保持：

- `portfolio_plan` 不成交。
- `paper_trading` 执行前仍做硬风控复核。
- `order_intent` 不能绕过现金、T+1、时间和数量规则。

## 14. 第三轮验收样例

执行：

```powershell
python .\paper_trading.py init --account paper_default --cash 100000
python .\paper_trading.py rollover --account paper_default --trade-date 2026-07-06
```

预期：

- 账户可用现金 100000。
- `last_rollover_trade_date = 2026-07-06`。

### 14.2 多候选排序

输入：

```text
002563 WATCH_SMALL MEDIUM score=79
600398 WAIT MEDIUM
601138 WATCH_SMALL LOW
```

预期：

- 002563 进入计划。
- 600398 只记录。
- 601138 因 LOW 降级为只记录或低优先级。

### 14.3 资金不足

账户：

```text
available_cash = 3000
total_assets = 100000
cash_reserve_pct = 20
```

预期：

```text
allocation_plan.status = NO_BUY_BUDGET
```

不生成可执行 `order_intent`。

### 14.4 单票超仓

已有持仓：

```text
symbol = 002563
position_pct = 29%
max_single_position_pct = 30%
```

预期：

- 新买入金额被大幅压缩。
- 不足 100 股时 `DEFERRED`。

### 14.5 非交易日

输入：

```text
time_context.session_name = NON_TRADING
```

预期：

- 可生成研究/计划报告。
- 不生成 `READY_FOR_CONFIRM` 的可执行 order_intent。
- 报告标注 `execution_allowed = false`。

## 16. 第三轮完成标准

第三轮代码完成后，应满足：

- 能读取多个 `decision_result`。
- 能找到对应 `strategy_snapshot`。
- 能生成 `risk_check_result`。
- `WAIT`、`DO_NOT_BUY`、`DATA_BLOCKED` 不进入买入计划。
- 非交易日不能生成可执行 intent。
- 未 rollover 不能生成可执行 intent。
- 现金不足不能生成可执行 intent。
- 单票超仓会缩小或拒绝。
- 多候选能按透明分数排序。
- 本金有限时能分配给优先级最高的候选。
- 生成的 `order_intent` 数量满足 100 股整数。
- 报告能解释买谁、买多少、为什么不买。
- 不产生任何 `paper_trade`。
