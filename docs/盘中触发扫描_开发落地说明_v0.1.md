# 盘中触发扫描开发落地说明 v0.1

> 本文档用于指导 `intraday` 盘中触发扫描的下一阶段开发。  
> 当前已完成第一版手动扫描实现。  
> 盘中触发扫描的目标是“发现触发、输出提醒、等待人工确认”，不是自动实盘下单。

---

## 1. 核心定位

盘中触发扫描是 `09_workflow` 的一部分。

它的定义是：

```text
盘中触发扫描 = 用盘前计划 + 当前价格，判断是否触发买入/卖出提醒
```

它不重新做完整策略判断。

盘中只做：

- 读取盘前计划。
- 读取触发价列表。
- 刷新当前价格。
- 判断价格是否触发。
- 快速风控校验。
- 输出触发事件和提醒报告。
- 等待人工确认。

盘中不做：

- 大量抓财报。
- 大量抓公告。
- 重算长期指标。
- 反复调用 AI 重新判断。
- 自动实盘下单。
- 用盘后数据影响盘中。

---

## 2. 为什么不能盘中重算复杂策略

A 股盘中时间很短，错误时间和错误数据会直接导致错误交易。

盘中复杂判断的问题：

- 数据源延迟会造成误触发。
- 财报和公告数据盘中不一定可靠。
- 盘中反复重算容易引入未来函数或不一致口径。
- AI 临时解释可能覆盖硬规则。
- 高频扫描会浪费资源。

所以第一版只做低频触发扫描：

```text
盘前生成计划
盘中只看触发
触发后人工确认
盘后再复盘
```

---

## 3. 第一版目标

第一版要实现：

1. 手动执行一次盘中扫描。
2. 读取当天 `trigger_price_list`。
3. 读取当天 `pre_market_plan`。
4. 刷新或读取当前行情。
5. 判断买入、卖出、止损、风险触发。
6. 生成 `trigger_event`。
7. 生成中文盘中报告。
8. 不生成模拟成交。
9. 不自动确认下单。

第一版不做：

- 自动轮询。
- 实时行情 websocket。
- 自动实盘下单。
- 自动模拟盘成交。
- 分钟级技术指标。
- 多线程全市场扫描。

## 4. 输入

### 4.1 盘前计划

来自第四轮工作流：

```text
data/workflows/pre_market_plan_<account>_<trade_date>.json
```

用途：

- 今日买入候选。
- 今日持仓复核结果。
- 今日组合计划。
- 今日是否允许交易。
- 关联 `decision_id`、`snapshot_id` 和 `order_intent`。

### 4.2 触发价列表

来自第四轮工作流：

```text
data/workflows/trigger_price_list_<account>_<trade_date>.json
```

用途：

- 买入突破价。
- 止损价。
- MA20 / MA60 参考价。
- 近 20 日低点。
- 风险提示价。

当前第一轮已有字段：

```text
reduce_trigger_price
clear_trigger_price
middle_trend_price
resistance_price
stop_loss_price
```

### 4.3 当前价格

第一版价格来源可以先复用：

```text
data/stock_json/stock_data_<code>.json.quote.price
```

后续再接入实时或准实时行情。

当前价格字段至少需要：

```text
symbol
trade_date
price
quote_time
source
is_limit_up
is_limit_down
is_suspended
```

如果没有 `quote_time`，必须在报告里标注数据时间不精确。

### 4.4 账户和持仓

来自：

```text
data/portfolio/accounts.json
data/portfolio/positions.json
```

用途：

- 判断可用现金。
- 判断可卖数量。
- 判断是否已经持仓。
- 判断 T+1 是否可卖。
- 判断单票仓位。

```text
data/intraday/trigger_events.jsonl
```

## 6. trigger_event 结构

每次触发写一条：

```json
{
  "trigger_event_id": "trig_20260706_093501_002563",
  "account_id": "paper_default",
  "trade_date": "2026-07-06",
  "scan_id": "scan_20260706_093501",
  "symbol": "002563",
  "name": "森马服饰",
  "event_type": "BUY_BREAKOUT",
  "trigger_price": 6.20,
  "current_price": 6.23,
  "price_source": "stock_json.quote.price",
  "quote_time": "2026-07-06T09:35:00+08:00",
  "decision_id": "dr_xxx",
  "snapshot_id": "ss_xxx",
  "source_decision_path": "data/decision_results/xxx.json",
  "source_plan_path": "data/workflows/pre_market_plan_xxx.json",
  "severity": "INFO",
  "suggested_action": "REMIND_BUY_CONFIRM",
  "execution_allowed": false,
  "requires_human_confirm": true,
  "risk_status": "NOT_CHECKED",
  "blocked_reason": null,
  "created_at": "2026-07-06T09:35:01+08:00"
}
```

重要原则：

- `trigger_event` 是提醒，不是订单。
- `execution_allowed` 第一版默认 `false`。
- 真正成交仍必须走 `paper_trading.py apply ... --confirm` 或后续显式确认流程。

## 8. 触发类型

第一版支持以下触发。

### 8.1 BUY_BREAKOUT

含义：

```text
当前价 >= resistance_price
```

适用：

- 买入候选。
- `order_intent.status = READY_FOR_CONFIRM`。

输出：

```text
REMIND_BUY_CONFIRM
```

注意：

- 不能自动买。
- 如果涨停，提示 `LIMIT_UP_RISK`。
- 如果没有盘前计划，不触发买入。

### 8.2 BUY_PULLBACK

第一版可以先预留，不默认启用。

含义：

```text
当前价接近 MA20 或预设回踩价
```

后续需要策略明确回踩买入规则后再启用。

### 8.3 SELL_STOP_LOSS

含义：

```text
当前价 <= stop_loss_price
```

适用：

- 当前持仓。
- 可卖数量 > 0。

输出：

```text
REMIND_SELL_CONFIRM
```

如果可卖数量为 0：

```text
NO_AVAILABLE_QUANTITY
```

### 8.4 SELL_BREAK_MA20

含义：

```text
当前价 < reduce_trigger_price
```

通常对应 MA20。

输出：

```text
REMIND_REDUCE_CONFIRM
```

### 8.5 SELL_BREAK_MA60

含义：

```text
当前价 < middle_trend_price
```

输出：

```text
REMIND_RISK_REVIEW
```

第一版不直接建议清仓，只提示中线趋势风险。

### 8.6 SELL_BREAK_LOW20

含义：

```text
当前价 < clear_trigger_price
```

通常对应近 20 日低点。

输出：

```text
REMIND_CLEAR_CONFIRM
```

### 8.7 LIMIT_UP_RISK

含义：

```text
当前涨停或接近涨停
```

买入侧：

```text
涨停买入可能成交失败，不自动追买
```

### 8.8 LIMIT_DOWN_RISK

含义：

```text
当前跌停或接近跌停
```

卖出侧：

```text
跌停卖出可能成交失败，只能提示风险
```

### 8.9 PRICE_TIME_MISMATCH

含义：

```text
行情 trade_date != 扫描 trade_date
```

处理：

```text
BLOCK_TRIGGER
```

## 9. 快速风控

盘中触发后的快速风控只检查执行硬条件。

买入检查：

```text
是否交易日
是否盘中
是否已执行 daily_rollover
是否有可用现金
是否超过单票上限
是否涨停
是否停牌
是否已经重复触发/重复执行
```

卖出检查：

```text
是否交易日
是否盘中
是否有持仓
是否有可卖数量
是否跌停
是否停牌
是否 T+1 受限
是否重复触发
```

第一版只输出风控状态，不自动生成订单：

```text
PASS_REMIND
WARN_REMIND
BLOCK_REMIND
```

## 10. 重复触发处理

同一账户、同一交易日、同一股票、同一触发类型不能无限重复提醒。

建议去重键：

```text
account_id + trade_date + symbol + event_type + decision_id
```

同一个触发已记录后，再次扫描：

- 默认不重复写 `trigger_event`。
- 报告中可以显示“已触发过”。

如果价格从未触发区重新触发区，第一版仍不重复提醒。

后续可以加入冷却时间：

```text
cooldown_minutes = 30
```

## 11. 命令设计

建议新增：

```powershell
python .\workflow.py intraday --account paper_default --trade-date 2026-07-06
```

可选参数：

```powershell
--plan-path data/workflows/pre_market_plan_paper_default_2026-07-06.json
--trigger-path data/workflows/trigger_price_list_paper_default_2026-07-06.json
--symbols 002563,600398
--allow-non-trading
```

说明：

- 默认自动按 `account + trade_date` 查找计划文件。
- `--symbols` 只扫描指定股票。
- `--allow-non-trading` 只用于调试，不生成执行建议。

## 12. 报告内容

盘中报告：

```text
data/reports/intraday_report_<account>_<trade_date>.md
```

必须包含：

- 当前扫描时间。
- 当前 session。
- 扫描股票数。
- 触发事件数。
- 买入触发。
- 卖出触发。
- 阻塞原因。
- 已触发但不重复提醒的事件。
- 数据时间不匹配。
- 下一步人工确认提示。

报告必须写清：

```text
本报告是盘中提醒，不是成交记录。
```

---

## 13. 数据准确性要求

盘中最容易出错的是价格时间。

必须检查：

```text
quote.trade_date == scan.trade_date
```

如果缺少 `quote_time`：

```text
data_time_precision = LOW
```

如果行情价格为 0、空、负数：

```text
PRICE_MISSING
```

如果当前价格明显异常，例如偏离昨收超过规则涨跌停范围：

```text
PRICE_ABNORMAL
```

第一版可以只做基础校验，后续再接完整涨跌停规则。

---

## 14. 与模拟盘执行的关系

第一版盘中扫描不直接成交。

触发后的模拟盘执行仍然需要人工选择具体 `decision_result`：

```powershell
python .\paper_trading.py apply <decision_path> --account paper_default --confirm
```

后续可以新增：

```powershell
python .\workflow.py confirm-trigger <trigger_event_id> --account paper_default
```

但该功能必须另写文档再开发。

---

## 15. 与真实交易的关系

第一版禁止实盘自动交易。

未来如果接券商或插件，盘中触发也必须经过：

```text
trigger_event
  ↓
快速风控
  ↓
人工确认
  ↓
订单预览
  ↓
二次确认
  ↓
下单
```

必须支持：

- 防重复下单。
- 急停开关。
- 账户对账。
- 成交回报。
- 异常撤单。

这些不属于第一版。

## 18. 验收标准

第一版完成后，应满足：

- 非交易日默认不扫描或只输出阻塞。
- 非盘中时段默认不输出执行提醒。
- 能读取当天盘前计划。
- 能读取当天触发价列表。
- 能读取当前价格。
- 价格交易日不一致时阻塞。
- 当前价突破买入价时生成 `BUY_BREAKOUT`。
- 当前价跌破止损价时生成 `SELL_STOP_LOSS`。
- 当前价跌破 MA20 / MA60 / LOW20 时生成对应风险事件。
- 没有可卖数量时卖出触发标记阻塞。
- 同一事件重复扫描不会重复写多条触发。
- 生成 `intraday_report`。
- 不自动生成成交。
- 不改变账户现金和持仓。

## 20. 方向边界

盘中模块最重要的是克制。

必须坚持：

```text
盘前决策
盘中触发
人工确认
盘后复盘
```

不要把盘中扫描做成：

- 临时追涨系统。
- AI 临时喊单系统。
- 高频交易系统。
- 自动实盘下单系统。

第一版的价值是减少漏看触发和减少情绪化操作，而不是提高交易频率。
