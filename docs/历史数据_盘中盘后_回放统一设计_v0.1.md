# 历史数据、盘中盘后与历史回放统一设计 v0.1

> 目标：把“每天固定历史数据”“盘中观察数据”“盘后收盘数据”“策略输入快照”“历史回放”放到同一套时间正确的数据体系里。
> 本文先定方向和边界，后续再拆开发落地文档和代码实现。

---

## 1. 核心判断

不要把系统简单拆成“两套库”：

```text
盘中库
盘后库
```

这样会导致：

- 策略逻辑要写两套。
- 回放难复用。
- 容易把盘后数据误用于盘中。
- 后面字段扩展和迁移 PostgreSQL 更麻烦。

更合理的设计是：

```text
统一数据模型
  + 明确数据时点
  + 明确数据生效时间
  + 明确数据用途
  + 明确策略输入快照
  + 明确回放可见数据边界
```

也就是：

```text
daily_market_snapshots      日终固定事实
intraday_observations       盘中观察点
strategy_snapshots          策略实际输入快照
historical_replay_runs      历史回放运行
replay_daily_states         回放每日状态
```

---

## 2. 为什么必须先做历史数据固定

当前 `market_quotes` 更像“最新行情摘要”。

它适合：

- 页面展示。
- 当前自选股列表。
- 当前盘后诊股。

但它不适合：

- 历史回放。
- 两个月后复盘今天的策略。
- 证明当时系统到底看到了什么。

原因是 `market_quotes` 会被覆盖。

如果今天 2026-07-08 的数据没有固定，未来再看 2026-07-08 时，系统可能看到的是后来接口重新计算、修正或污染过的数据。

所以必须新增不可覆盖或可审计更新的历史表。

## 3. 数据层分工

### 3.1 current 层：当前最新摘要

表：

```text
market_quotes
```

用途：

- 当前页面展示。
- 当前自选股列表。
- 当前持仓估值。

特点：

- 一只股票只保留最新摘要。
- 可以覆盖更新。
- 不作为历史回放事实源。

### 3.2 daily 层：日终历史快照

建议表：

```text
daily_market_snapshots
```

用途：

- 固定每日收盘行情。
- 固定每日估值摘要。
- 固定每日技术指标。
- 作为历史回放和盘后诊股的事实源。

特点：

- 一只股票一个交易日一条。
- 以 `symbol + trade_date` 唯一。
- 盘后写入。
- 默认不允许无理由覆盖。
- 如需重跑，必须记录批次和覆盖原因。

### 3.3 intraday 层：盘中观察点

建议表：

```text
intraday_observations
```

用途：

- 盘中盯盘。
- 止损触发。
- 突破/跌破关键价。
- 模拟成交参考。
- 盘中异常记录。

特点：

- 一只股票一天可以多条。
- 每条必须有 `observed_at`。
- 不能当作收盘价。
- 第一版只记录持仓、明日关注池、盘中盯盘股票。
- 第一版不要全市场分钟级采集。

### 3.4 strategy 层：策略输入快照

建议表：

```text
strategy_snapshots
strategy_snapshot_items
```

用途：

- 保存某次决策实际使用的数据。
- 决策结果必须引用对应 `snapshot_id`。
- 历史回放时也必须生成当时可见的 `strategy_snapshot`。

特点：

- 策略引擎只读 `strategy_snapshot`。
- 不允许策略临时回头读接口。
- 不允许策略直接读未来数据。
- 这是解释“当时为什么这么判断”的核心证据。

### 3.5 replay 层：历史回放

建议表：

```text
historical_replay_runs
replay_daily_states
replay_orders
replay_trades
replay_positions
replay_metrics
```

用途：

- 固定策略版本。
- 固定回放样本。
- 固定每个交易日可见数据。
- 输出策略表现、错误案例和调参记录。

特点：

- 回放不是直接读最新行情。
- 回放必须按交易日推进。
- 每一天只能看当时已经生效的数据。

## 4. 盘中和盘后的区别

### 4.1 盘中数据

盘中数据是“观察点”，不是最终事实。

例子：

```text
2026-07-08 10:15:03
000100
price = 5.31
pct_change = 0.96%
observed_at = 2026-07-08T10:15:03+08:00
session_type = INTRADAY
```

盘中数据可以用于：

- 是否跌破止损。
- 是否触发买入价。
- 是否出现异常波动。
- 是否生成盘中提醒。

盘中数据不应该用于：

- 计算当天最终日线。
- 生成当天盘后复盘结论。
- 回填日终历史数据。
- 替代收盘价。

### 4.2 盘后数据

盘后数据是“日终事实”。

例子：

```text
2026-07-08
000100
close = 5.26
pct_change = 1.35%
ma20 = 5.12
ma60 = 4.57
session_type = POST_MARKET
effective_from = 2026-07-08T15:00:00+08:00 或实际确认时间
```

盘后数据可以用于：

- 当天盘后诊股。
- 生成明日关注池。
- 次日盘前计划。
- 历史回放。
- 模拟盘每日结算。

盘后数据不能用于：

- 当天盘中交易判断。
- 当天盘中触发扫描。

## 6. 全市场盘后采集

用户目标：

```text
收盘后用 1-2 小时，把几千只 A 股当天数据固定下来。
```

这是合理目标，但必须分阶段。

### 6.1 第一阶段

先支持：

- 持仓。
- 自选股。
- 明日关注池。

写入：

```text
market_quotes
daily_market_snapshots
market_data_batches
```

### 6.2 第二阶段

支持全市场：

- 读取 A 股代码池。
- 分批采集。
- 限速。
- 重试。
- 断点续跑。
- 失败列表。
- 数据日期校验。
- 异常值校验。

### 6.3 全市场采集不要影响诊股

全市场采集是数据准备，不是策略判断。

流程应该是：

```text
盘后全市场采集
  -> 写 daily_market_snapshots
  -> 数据质量检查
  -> 盘后诊股读取固定数据
  -> 明日关注池
```

## 7. 历史回放如何使用这些数据

历史回放不能直接读“最新数据”。

正确流程：

```text
选择回放区间
  ↓
按交易日从 start_date 推进到 end_date
  ↓
每个交易日构建当时可见数据
  ↓
生成 REPLAY strategy_snapshot
  ↓
运行策略引擎
  ↓
生成模拟订单/成交
  ↓
更新回放账户、持仓、现金
  ↓
写 replay_daily_states
  ↓
输出 replay_metrics
```

### 7.1 盘前回放口径

D 日盘前只能使用：

```text
trade_date <= D-1 的 daily_market_snapshots
已披露且 effective_from <= D 盘前决策时间的财报/公告
上一交易日盘后账户和持仓快照
```

不能使用：

```text
D 日收盘价
D 日盘后公告
D 日尚未发生的盘中价格
```

### 7.2 盘中回放口径

D 日盘中只能使用：

```text
D 日当前 observed_at 之前的 intraday_observations
D-1 日终数据
盘前计划和触发价
```

不能使用：

```text
D 日收盘后才能确认的日线指标
observed_at 之后的价格
```

### 7.3 盘后回放口径

D 日盘后可以使用：

```text
D 日 daily_market_snapshots
D 日收盘后可见指标
D 日盘后账户和持仓快照
```

但这些数据只能影响：

```text
D 日盘后复盘
D+1 日盘前计划
```

不能反向影响：

```text
D 日盘中决策
D 日模拟成交
```

## 8. 策略运行口径

策略不直接区分“盘中策略表”和“盘后策略表”。

策略区分的是：

```text
session_type
task_type
available_data
rule_set
```

### 8.1 盘中策略

盘中策略第一版只做：

- 止损触发。
- 证伪价触发。
- 突破买点提醒。
- 跌破关键位提醒。
- 涨跌停风险提醒。
- 模拟成交条件检查。

盘中策略不做：

- 完整基本面重算。
- 全市场诊股。
- 根据盘中噪声频繁改长期结论。

### 8.2 盘后策略

盘后策略做：

- 固定日终数据。
- 诊断持仓风险。
- 诊断自选股买入候选。
- 生成明日关注池。
- 更新模拟盘日终净值。
- 记录账户和持仓快照。
- 复盘策略错误。

### 8.3 历史回放策略

历史回放策略做：

- 按历史当时视角运行同一套规则。
- 不使用未来数据。
- 输出可统计结果。
- 暴露策略问题、数据问题、执行问题。

## 9. 数据量和 SQLite 可行性

全市场日线数据量估算：

```text
5000 只股票 × 250 交易日 = 125 万条 / 年
5 年约 625 万条
```

SQLite 短中期可以承受，前提是：

- 核心字段列化。
- 大 JSON 放 `payload_json`，不要每次页面查询都解析。
- 建好 `symbol + trade_date`、`trade_date` 索引。
- 批量写入使用事务。
- 页面必须分页。
- 不做全表无索引扫描。

未来迁移 PostgreSQL 的触发条件：

- 数据超过千万级且查询变慢。
- 多人同时使用。
- 需要远程部署。
- 后台任务持续写入，前端持续查询。
- 需要更强备份、权限和并发能力。

## 10. 字段扩展原则

后面策略一定会加字段，所以不能把所有字段一开始都做成硬列。

原则：

```text
高频查询字段 -> 列
低频解释字段 -> payload_json
稳定后高频使用 -> 再拆表
```

第一版列化：

- 代码。
- 交易日。
- 开高低收。
- 涨跌幅。
- 成交量。
- 成交额。
- PE/PB。
- MA20/MA60。
- 20 日涨幅。
- 数据来源。
- 采集时间。
- 生效时间。
- 数据质量状态。

先放 JSON：

- 北向资金。
- 融资余额。
- 股东户数。
- 行业数据。
- 公告摘要。
- 更多技术指标。
- 接口原始字段。

## 11. 第一版开发顺序

建议顺序：

```text
1. 新增 market_data_batches
2. 新增 daily_market_snapshots
3. 改盘后数据准备：双写 market_quotes + daily_market_snapshots
4. 盘后诊股优先读取 daily_market_snapshots
5. 增加历史数据页面：按交易日查看采集情况
6. 增加全市场盘后采集任务
7. 新增 intraday_observations
8. 盘中触发扫描写入 intraday_observations
9. 增强 strategy_snapshots 入库和引用
10. 历史回放改为读取 daily_market_snapshots 构建可见快照
```

不要先做：

- 全市场盘中轮询。
- 高频分钟级数据库。
- 自动实盘下单。
- 复杂调度系统。

## 12. 验收标准

### 12.1 历史数据固定

- 同一股票同一交易日只有一条日终快照。
- 重复采集不会产生重复数据。
- 能看到每次采集批次、成功数、失败数。
- 能列出缺数据股票。
- 能追溯某条快照来自哪个批次。

### 12.2 盘中盘后边界

- 盘中观察点不会覆盖日终快照。
- 盘后快照不会影响当天盘中决策。
- 策略输入快照必须记录 `session_type`。
- 决策结果必须能找到对应 `strategy_snapshot`。

### 12.3 历史回放

- 回放按交易日推进。
- D 日盘前不能使用 D 日收盘数据。
- D 日盘中不能使用 observed_at 之后的数据。
- D 日盘后数据只能影响 D+1。
- 每日回放状态可追溯。

## 13. 当前结论

下一步最值得做的是：

```text
daily_market_snapshots + market_data_batches
```

先把每天的历史数据固定下来。

然后再做：

```text
盘后数据准备双写
盘后诊股读取日终快照
历史回放读取日终快照
```

最后再补：

```text
intraday_observations
盘中触发扫描
虚拟交易自动化
```

这个顺序更稳，因为数据一旦没有固定，未来回放和模拟盘都会失真。

## 14. 历史数据回补

历史数据可以补，但必须和“从今天开始每天固定的数据”区分开。

### 14.1 两种历史数据

第一种：

```text
LIVE_CAPTURE
```

含义：

- 系统在当天盘后实际采集到的数据。
- 有真实的 `observed_at`。
- 能证明“当时系统已经看到这些数据”。
- 可信度最高。

第二种：

```text
BACKFILL
```

含义：

- 事后从接口或文件补回来的历史数据。
- 可以用于策略研究和技术回放。
- 不能证明当时系统真的已经看到这些数据。
- 必须显式标记。

第三种：

```text
MANUAL_IMPORT
```

含义：

- 人工导入或人工修正的数据。
- 必须记录来源、导入时间和原因。

### 14.2 为什么要区分

如果不区分，会出现严重问题：

- 用事后修正过的数据证明当时策略有效。
- 回放看起来很好，但真实当天并不可用。
- 接口后来补齐或修正了历史数据，导致历史现场被污染。
- 无法判断策略成功是因为规则好，还是因为使用了事后数据。

所以：

```text
LIVE_CAPTURE != BACKFILL
```

两者可以放在同一张表里，但必须通过字段区分。

### 14.3 daily_market_snapshots 需要支持的回补字段

建议字段：

```text
data_origin          LIVE_CAPTURE / BACKFILL / MANUAL_IMPORT
is_backfilled        0 / 1
backfilled_at        事后补录时间
backfill_batch_id    回补批次
source_version       数据源或接口版本
observed_at          系统获得该数据的时间
effective_from       理论上可被策略使用的时间
quality_status       OK / STALE / PARTIAL / ERROR
quality_flags_json   缺失、异常、冲突等质量标记
```

对于 `LIVE_CAPTURE`：

```text
data_origin = LIVE_CAPTURE
is_backfilled = 0
observed_at = 当天盘后实际采集时间
effective_from = 当天收盘后数据确认时间
```

对于 `BACKFILL`：

```text
data_origin = BACKFILL
is_backfilled = 1
observed_at = 补录发生时间
backfilled_at = 补录发生时间
effective_from = trade_date 收盘后
```

注意：

`effective_from` 表示这条数据理论上属于哪个历史时点可用，不等于系统当时真的采集到了它。

### 14.4 回补流程

```text
选择时间范围
  ↓
选择股票范围
  ↓
创建 BACKFILL_DAILY 批次
  ↓
分批拉取历史日线
  ↓
写入 daily_market_snapshots
  ↓
标记 data_origin = BACKFILL
  ↓
执行质量校验
  ↓
记录缺失、异常、冲突
  ↓
生成回补报告
```

股票范围：

```text
单票
自选股 tab
持仓历史股票
全市场
```

时间范围：

```text
start_date
end_date
```

批次类型：

```text
market_data_batches.batch_type = BACKFILL_DAILY
```

### 14.5 回补质量校验

回补后必须检查：

- 是否缺交易日。
- 是否缺股票。
- 是否价格为 0 或空。
- 是否成交量异常。
- 是否涨跌幅异常。
- 是否重复。
- 是否和已有 `LIVE_CAPTURE` 冲突。
- 是否数据源日期和目标 `trade_date` 不一致。

异常不应静默忽略。

应写入：

```text
quality_status
quality_flags_json
data_conflicts
```

### 14.6 冲突处理

如果同一只股票同一交易日已经有数据，优先级建议：

```text
LIVE_CAPTURE > MANUAL_IMPORT > BACKFILL
```

规则：

- 回补数据不能自动覆盖 `LIVE_CAPTURE`。
- 回补数据和 `LIVE_CAPTURE` 不一致时，写入冲突记录。
- 页面提示冲突。
- 只有人工确认后才允许修正。
- 修正必须保留原值、修正值、修正原因和修正时间。

建议后续增加表：

```text
data_conflicts
```

字段：

```text
conflict_id
table_name
symbol
trade_date
field_name
existing_value
incoming_value
existing_origin
incoming_origin
batch_id
status              OPEN / ACCEPT_EXISTING / ACCEPT_INCOMING / MANUAL_FIXED
created_at
resolved_at
resolution_note
```

### 14.7 历史回放模式

历史回放必须区分两种模式。

#### STRICT 严格模式

只允许使用：

```text
data_origin = LIVE_CAPTURE
```

用途：

- 验证系统真实历史表现。
- 验证“当时系统实际能不能做出这个判断”。

限制：

- 数据少。
- 从今天开始才逐步积累。
- 早期无法覆盖很长历史。

#### RESEARCH 研究模式

允许使用：

```text
data_origin IN (LIVE_CAPTURE, BACKFILL, MANUAL_IMPORT)
```

用途：

- 快速测试策略。
- 验证技术规则。
- 评估大样本表现。

限制：

- 报告必须标记“使用事后回补数据”。
- 不能当作系统当时真实可见能力证明。

### 14.8 回放报告必须标注数据来源

历史回放报告必须包含：

```text
replay_mode
LIVE_CAPTURE 数据条数
BACKFILL 数据条数
MANUAL_IMPORT 数据条数
缺失数据条数
冲突数据条数
是否存在未来函数风险
```

如果使用 `BACKFILL`，报告必须写明：

```text
本次回放使用事后回补数据，只适合策略研究，不代表当时系统真实可见。
```

### 14.9 当前建议

第一步不要先大规模回补多年全市场。

建议顺序：

```text
1. 先做 LIVE_CAPTURE：从今天开始每天固定。
2. 小范围 BACKFILL：先补持仓和自选股近 1 年。
3. 用 RESEARCH 模式验证策略链路。
4. 再扩大全市场 BACKFILL。
5. STRICT 模式随着每日采集逐步积累真实样本。
```

这样能避免一开始就陷入大规模历史数据清洗，同时不会耽误每天真实数据沉淀。
