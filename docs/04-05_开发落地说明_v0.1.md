# 04-05 开发落地说明 v0.1

> 本文档把 04 回测与模拟盘、05 持仓与资金系统翻译成第二轮可开发的最小闭环。  
> 当前仍然只写设计，不写代码。  
> 第二轮目标不是做完整回测系统，而是先让 `decision_result` 能进入模拟账户、持仓账本和每日快照。

---

## 1. 第二轮开发目标

第二轮只做一个最小闭环：

```text
decision_result
  ↓
paper_signal
  ↓
manual_confirm / conservative_policy
  ↓
paper_order
  ↓
paper_trade
  ↓
portfolio_account
  ↓
position / cash ledger
  ↓
daily_snapshot
  ↓
paper_report
```

第二轮要验证：

- 策略建议和模拟执行能分开。
- 模拟盘能维护现金、持仓、成本和可卖数量。
- A 股 100 股买入、T+1、可卖数量能被执行。
- 买入不能超过可用现金。
- 卖出不能超过可卖数量。
- 每笔模拟成交能追溯到 `decision_id` 和 `snapshot_id`。
- 每日账户和持仓快照能生成。
- 资金流水和持仓流水能复盘。

第二轮不做：

- 自动实盘下单。
- 券商接口同步。
- 严谨历史点时回测。
- 分钟级撮合。
- 组合排序和行业集中度。
- Web 界面。
- SQLite 正式 schema。

说明：

第一版先用 JSON 文件账本，等流程稳定后再进入 08 数据库设计。

## 3. 第二轮处理流程

### 3.1 总流程

```text
load_decision_result(decision_path)
  ↓
load_or_create_paper_account(account_id)
  ↓
load_positions(account_id)
  ↓
build_portfolio_context(account, positions)
  ↓
convert_decision_to_signal(decision_result)
  ↓
convert_signal_to_order(signal, portfolio_context)
  ↓
validate_order(order, account, position)
  ↓
simulate_fill(order, market_price, cost_model)
  ↓
apply_trade_to_portfolio(trade)
  ↓
write_ledgers()
  ↓
write_snapshots()
```

### 3.2 第二轮模式

第二轮支持四类命令形态：

```text
paper-init       初始化模拟账户
paper-rollover   交易日盘前换日，处理 T+1 解锁和当日额度重置
paper-apply      把某个 decision_result 应用到模拟盘
paper-snapshot   生成当日账户和持仓快照
```

建议命令：

```powershell
python -m ai_trader.paper_trading init --account paper_default --cash 100000
python -m ai_trader.paper_trading rollover --account paper_default --trade-date 2026-07-06
python -m ai_trader.paper_trading apply data/decision_results/decision_result_002563_sell_20260704T100000.json --account paper_default
python -m ai_trader.paper_trading snapshot --account paper_default
```

具体命令可在开发时调整，但输入输出契约不能变。

## 4. 开发模块建议

第二轮建议新增模块：

```text
src/ai_trader/accounts.py
src/ai_trader/portfolio.py
src/ai_trader/paper_trading.py
src/ai_trader/cost_model.py
src/ai_trader/performance.py
```

### 4.1 accounts.py

职责：

- 创建模拟账户。
- 读取账户状态。
- 更新现金。
- 生成账户快照。

不负责：

- 判断买卖。
- 生成订单。
- 计算策略结论。

### 4.2 portfolio.py

职责：

- 读取当前持仓。
- 更新持仓数量、成本、可卖数量。
- 处理 T+1 解锁。
- 生成持仓快照。
- 写入持仓流水。

不负责：

- 策略判断。
- 组合排序。
- 行业集中度。

### 4.3 paper_trading.py

职责：

- 把 `decision_result` 转成模拟信号。
- 把模拟信号转成模拟订单。
- 校验订单。
- 生成模拟成交。
- 调用 `accounts` 和 `portfolio` 更新账本。

不负责：

- 重新判断股票是否该买卖。
- 覆盖风控结论。

### 4.4 cost_model.py

职责：

- 计算佣金。
- 计算印花税。
- 计算滑点。
- 计算买入所需现金。
- 计算卖出净回款。

### 4.5 performance.py

第二轮只做最小绩效：

- 总资产。
- 累计收益率。
- 单日收益率。
- 最大回撤。
- 持仓数量。
- 权益仓位。

胜率、盈亏比、年化收益率可以后移。

## 5. 文件账本契约

### 5.0 标的和行情基础信息

第二轮先不单独建设完整行情库，但必须把数据归属设计清楚。

基础归属：

```text
instruments      标的基础信息，一只股票一条
market_quotes    行情主数据，一个交易日或观察时点一条
positions        当前持仓，一只当前持仓一条
position_snapshots 每日持仓快照，每天每只持仓一条
```

第二轮文件账本可以先不创建 `instruments.json` 和 `market_quotes.jsonl`，因为第一轮已经有：

```text
data/stock_json/stock_data_<code>.json
```

其中：

```text
quote      提供当前价、交易日、涨跌幅、估值等
meta       提供部分标的信息
technical  提供均线和技术价位
```

但实现时必须遵守：

- `stock_data_<code>.json.quote` 是第二轮临时行情输入。
- `positions.json.market_price` 只是当前估值价格副本。
- `position_snapshots.jsonl.market_price` 是当日快照价格副本。
- 不能把 `positions.json` 当作行情主数据源。

后续进入 08 数据库设计时，应把它们拆成：

```text
instruments
market_quotes
technical_indicators
valuation_snapshots
positions
position_snapshots
```

#### 第二轮最小行情输入

`paper-apply` 至少需要能拿到：

```text
symbol
name
trade_date
price
```

来源优先级：

```text
strategy_snapshot.quote.price
  ↓
decision_result.reference_price
  ↓
decision_result.quote.price
  ↓
data/stock_json/stock_data_<code>.json.quote.price
```

说明：

- 当前第一轮 `decision_result` 不保证包含 `quote`。
- 第二轮必须优先读取与 `decision_result.snapshot_id` 对应的 `strategy_snapshot.quote.price`。
- 如果使用 `data/stock_json/stock_data_<code>.json.quote.price` 作为兜底价格，必须校验 `quote.trade_date` 与 `decision_result.time_context.trade_date` 一致。
- 如果交易日不一致，不能成交，只能拒绝或记录信号。

如果价格缺失：

```text
order.status = REJECTED
reject_reason = PRICE_MISSING
```

如果价格交易日不一致：

```text
order.status = REJECTED
reject_reason = PRICE_TIME_MISMATCH
```

#### 快照必须复制行情价格

`position_snapshot` 必须保存当日用于估值的：

```text
market_price
quote_time
quote_source
market_value
unrealized_pnl
unrealized_pnl_pct
position_pct
```

原因是后续行情会变化，复盘不能依赖“最新价格”还原过去的持仓状态。

### 5.3 position_locks.jsonl

`position_locks` 记录 T+1 锁定明细。

第二轮不能只依赖 `positions.json.locked_quantity` 聚合数量，因为聚合数量无法判断每一笔买入是否已经到解锁日期。

每次买入成交必须追加一条锁定记录：

```text
lock_id
account_id
symbol
buy_trade_id
buy_trade_date
unlock_trade_date
locked_quantity
remaining_locked_quantity
status              OPEN / RELEASED
created_at
released_at
```

规则：

- `unlock_trade_date` 必须来自交易日历，不能按自然日加一天。
- 没有完整交易日历时，必须由命令显式传入解锁交易日。
- `positions.json.locked_quantity` 是当前聚合结果，必须能由未释放的 `position_locks` 汇总校验。
- 如果聚合结果和锁定明细不一致，暂停交易并要求人工修正。

### 5.4 JSONL 流水

流水文件使用 JSON Lines。

每行是一条独立记录，方便追加：

```json
{"cash_ledger_id":"cl_20260704_000001","account_id":"paper_default","event_type":"BUY_OUTFLOW","amount":5560.0}
```

禁止把 JSONL 写成一个大数组，避免每次追加都重写全文件。

## 6. decision_result 到模拟信号

### 6.1 paper_signal

模拟信号只记录策略建议，不等于订单。

字段：

```text
signal_id
account_id
decision_id
snapshot_id
symbol
name
task_type
final_action
confidence
signal_action
signal_quantity
signal_cash_amount
source_decision_time
source_decision_path
source_decision_hash
decision_schema_version
strategy_version
action_reason
position_plan_snapshot
created_at
status
blocked_reason
```

`status`：

```text
RECORDED
ORDER_CREATED
BLOCKED
IGNORED
```

追溯要求：

- `paper_signal` 必须复制 `final_action`、`confidence`、`action_reason` 和 `position_plan` 的关键内容。
- `source_decision_path` 记录原始 `decision_result` 文件路径。
- `source_decision_hash` 记录原始 `decision_result` 文件内容哈希。
- 即使第一轮历史文件被清理，模拟盘仍能复盘当时为什么记录或执行这条信号。

### 6.2 action 映射

买入类：

| `final_action` | 第二轮默认处理 |
|---|---|
| `BUY` | 可生成模拟买入订单，但需要通过资金校验 |
| `WATCH_SMALL` | 可生成观察仓买入订单 |
| `WAIT` | 只记录信号 |
| `DO_NOT_BUY` | 只记录信号 |
| `DATA_BLOCKED` | 只记录信号，标记阻塞 |

卖出类：

| `final_action` | 第二轮默认处理 |
|---|---|
| `HOLD` | 只记录信号 |
| `REDUCE_HALF` | 生成卖出可卖数量 50% 的订单 |
| `REDUCE_TO_WATCH` | 生成减仓到观察仓的订单 |
| `CLEAR` | 生成卖出全部可卖数量的订单 |
| `NO_SELL_T_PLUS` | 只记录信号，标记 T+1 阻塞 |
| `PRE_EVALUATION` | 只记录信号，不生成订单 |
| `DATA_BLOCKED` | 只记录信号，标记阻塞 |

### 6.3 默认保守策略

第二轮建议默认：

```text
auto_order = false
```

也就是先记录信号，不自动生成订单。

如果命令显式传入：

```text
--confirm
```

才允许从信号生成模拟订单。

这样可以避免策略还没验证时模拟盘被过早污染。

## 7. 模拟订单契约

### 7.1 paper_order

字段：

```text
order_id
account_id
signal_id
decision_id
snapshot_id
symbol
name
side                  BUY / SELL
order_type            MARKET / LIMIT
requested_quantity
limit_price
reference_price
status
reject_reason
created_at
updated_at
```

`status`：

```text
PENDING
FILLED
REJECTED
CANCELLED
```

第二轮不做部分成交，暂不支持 `PARTIALLY_FILLED`。

### 7.2 买入数量规则

买入数量必须满足：

```text
quantity % 100 == 0
quantity > 0
```

建议数量来源优先级：

```text
decision_result.position_plan.suggested_quantity
  ↓
decision_result.position_plan.suggested_cash_amount / reference_price
  ↓
默认观察仓金额 / reference_price
```

计算后向下取整到 100 股：

```text
quantity = floor(raw_quantity / 100) * 100
```

不足 100 股：

```text
order.status = REJECTED
reject_reason = CASH_NOT_ENOUGH_FOR_ONE_LOT
```

### 7.3 卖出数量规则

卖出数量不能超过：

```text
available_quantity
```

第二轮卖出数量：

```text
REDUCE_HALF     floor(available_quantity * 0.5)
REDUCE_TO_WATCH max(available_quantity - target_watch_quantity, 0)
CLEAR           available_quantity
```

A 股卖出数量必须经过归一化：

```text
normalize_sell_quantity(action, raw_quantity, available_quantity)
```

规则：

- `CLEAR` 清仓时，允许卖出全部 `available_quantity`，即可以包含不足 100 股的尾数。
- 非清仓卖出时，卖出数量必须向下取整到 100 股整数倍。
- 非清仓卖出归一化后不足 100 股时，不生成订单。
- 不允许把不足 100 股的尾数拆成多次卖出。

示例：

```text
available_quantity = 1000
REDUCE_HALF raw = 500
normalized = 500 向下取整到 100 股整数倍，即 500

available_quantity = 1100
REDUCE_HALF raw = 550
normalized = 500

available_quantity = 550
CLEAR normalized = 550
```

如果卖出数量不符合规则：

```text
order.status = REJECTED
reject_reason = SELL_QUANTITY_NOT_ALLOWED
```

## 8. 模拟成交契约

### 8.1 成交时点

第二轮默认使用：

```text
SAME_REFERENCE
```

含义：

- 不做严谨回测。
- 使用 `strategy_snapshot.quote.price` 或 `decision_result.reference_price` 作为参考价。
- 报告必须标注“粗略模拟成交，不代表真实可成交”。

成交前必须检查 `time_context`：

```text
decision_result.time_context.is_trading_day == true
decision_result.time_context.session_name != NON_TRADING
```

如果不是交易日：

```text
只写入 paper_signal
不生成 paper_order
不生成 paper_trade
signal.status = BLOCKED
blocked_reason = NON_TRADING_DAY
```

说明：

- 周末、节假日生成的决策报告只能作为计划参考，不能直接进入模拟成交。
- 盘后生成的报告如果使用 `SAME_REFERENCE`，也必须在报告中明确标注“估值模拟，不代表真实可成交”。
- 后续如果需要隔夜计划单，应单独设计 `planned_order`，不要把计划单伪装成已成交订单。

后续进入历史回放时再切换：

```text
NEXT_OPEN
```

### 8.2 成交价格

买入：

```text
fill_price = reference_price * (1 + slippage_rate)
```

卖出：

```text
fill_price = reference_price * (1 - slippage_rate)
```

滑点口径：

```text
gross_amount = quantity * fill_price
slippage_cost = abs(fill_price - reference_price) * quantity
```

其中：

- `fill_price` 已经包含滑点。
- `slippage_cost` 只用于报告展示和复盘归因。
- `net_amount` 不能再次加减 `slippage_cost`。
- 禁止滑点既体现在成交价里，又再次进入现金流水。

默认参数：

```text
commission_rate = 0.0003
min_commission = 5
stamp_tax_rate_sell = 0.0005
slippage_rate = 0.001
```

### 8.3 paper_trade

字段：

```text
trade_id
order_id
account_id
decision_id
snapshot_id
symbol
name
side
quantity
reference_price
fill_price
gross_amount
commission
stamp_tax
slippage_cost
net_amount
trade_time

```

买入：

```text

```

卖出：

```text

```

## 9. 持仓和资金更新规则

### 9.1 买入成交

现金：

```text
available_cash -= net_amount
today_buy_used += net_amount
```

持仓：

```text
total_quantity += quantity
locked_quantity += quantity
available_quantity 不增加
```

平均成本：

```text
new_avg_cost =
  (old_quantity * old_avg_cost + net_amount)
  / (old_quantity + quantity)
```

流水：

```text
cash_ledger: BUY_OUTFLOW
position_ledger: BUY
```

### 9.2 卖出成交

现金：

```text
available_cash += net_amount
today_sell_amount += gross_amount
```

持仓：

```text
total_quantity -= quantity
available_quantity -= quantity
locked_quantity 不变
avg_cost 不变
```

如果 `total_quantity = 0`：

```text
position_status = CLOSED
```

流水：

```text
cash_ledger: SELL_INFLOW
position_ledger: SELL
```

### 9.3 T+1 解锁

买入成交后，除更新 `positions.json.locked_quantity` 外，还必须写入 `position_locks.jsonl`：

```text
buy_trade_date = 成交交易日
unlock_trade_date = 下一交易日
remaining_locked_quantity = bought_quantity
status = OPEN
```

盘前解锁时，只能释放已到期锁定明细：

```text
unlockable = sum(position_locks.remaining_locked_quantity
                 where status = OPEN
                 and unlock_trade_date <= current_trade_date)

available_quantity += unlockable
locked_quantity -= unlockable
position_locks.status = RELEASED
position_locks.remaining_locked_quantity = 0
```

流水：

```text
position_ledger: T_PLUS_UNLOCK
```

第二轮如果没有完整交易日历，允许手动执行：

```powershell
python -m ai_trader.paper_trading rollover --account paper_default --trade-date 2026-07-06
```

但必须满足：

- `--trade-date` 是交易日。
- `--trade-date` 大于买入成交日。
- 只能解锁 `unlock_trade_date <= --trade-date` 的锁定明细。
- 报告中必须标注“手动解锁”。

禁止：

- 把全部 `locked_quantity` 无条件解锁。
- 在买入当天手动解锁。
- 在周末或节假日按自然日解锁。

### 9.4 每日换日处理

每个交易日盘前必须执行 `daily_rollover`。

职责：

```text
处理到期 T+1 解锁
重置 today_buy_used
重置 today_sell_amount
刷新 time_context.trade_date
记录 rollover 日志
```

如果没有执行 `daily_rollover`，当天不允许生成新的模拟成交，只能记录信号。

## 10. 账户快照和持仓快照

### 10.1 account_snapshot

字段：

```text
snapshot_id
account_id
trade_date
snapshot_time
available_cash
frozen_cash
market_value
total_assets
daily_pnl
daily_return_pct
total_return_pct
max_drawdown_pct
equity_position_pct
cash_pct
position_count
created_at
```

计算：

```text
market_value = sum(position.market_value)
total_assets = available_cash + frozen_cash + market_value
total_return_pct = total_assets / initial_cash - 1
equity_position_pct = market_value / total_assets
cash_pct = available_cash / total_assets
```

### 10.2 position_snapshot

字段：

```text
snapshot_id
account_id
symbol
trade_date
snapshot_time
total_quantity
available_quantity
locked_quantity
avg_cost
market_price
market_value
unrealized_pnl
unrealized_pnl_pct
position_pct
buy_logic
invalidation_point
stop_loss_price
created_at
```

快照口径：

```text
positions.json              当前持仓，一只股票一条当前状态
position_snapshots.jsonl    每日持仓快照，每个交易日每只当前持仓一条记录
```

`position_snapshot` 不是“一个券一条历史数据”，而是“每天保存一次当日状态”。

如果账户当天持有 5 只股票，盘后就应该写入 5 条 `position_snapshot`。

如果当天没有任何交易，但仍持有 5 只股票，盘后仍然写入 5 条 `position_snapshot`。

建议唯一键：

```text
account_id + symbol + trade_date + snapshot_type
```

第二轮默认：

```text
snapshot_type = POST_MARKET
```

### 10.3 快照生成规则

第二轮至少支持手动生成：

```powershell
python -m ai_trader.paper_trading snapshot --account paper_default
```

盘后工作流接入后，再由 `post-market` 自动生成。

生成规则：

- 每天至少生成一条 `account_snapshot`。
- 对账户当日全部未清仓持仓逐只生成 `position_snapshot`。
- 不能只给当天有成交的股票生成持仓快照。
- 清仓当天可以生成一条数量为 0、状态为 `CLOSED` 的最终快照，方便复盘。

## 11. 异常与拒绝规则

### 11.1 订单拒绝

订单拒绝原因：

```text
NO_ACCOUNT
NO_POSITION
CASH_NOT_ENOUGH
CASH_NOT_ENOUGH_FOR_ONE_LOT
QUANTITY_NOT_LOT_SIZE
NO_AVAILABLE_QUANTITY
T_PLUS_ONE_BLOCKED
PRICE_MISSING
PRICE_TIME_MISMATCH
NON_TRADING_DAY
DATA_BLOCKED
PRE_EVALUATION_NOT_TRADABLE
ACTION_NOT_TRADABLE
SELL_QUANTITY_NOT_ALLOWED
LOCK_NOT_MATURED
DAILY_ROLLOVER_MISSING
```

### 11.2 数据异常

出现以下情况必须暂停执行：

- `available_cash < 0`
- `total_assets <= 0`
- `total_quantity < 0`
- `available_quantity < 0`
- `locked_quantity < 0`
- `available_quantity + locked_quantity > total_quantity`
- `sum(open position_locks.remaining_locked_quantity) != locked_quantity`
- `avg_cost <= 0` 且 `total_quantity > 0`
- `reference_price` 缺失或小于等于 0

### 11.3 只记录不执行

以下结论只能记录信号，不生成订单：

```text
WAIT
DO_NOT_BUY
DATA_BLOCKED
HOLD
NO_SELL_T_PLUS
PRE_EVALUATION
```

除非后续有非常明确的人工覆盖机制，否则第二轮不允许这些动作下单。

### 12.2 记录 PRE_EVALUATION

输入：

```text
decision_result.final_action = PRE_EVALUATION
```

预期：

- 写入 `signals.jsonl`。
- 不生成订单。
- 不改变现金。
- 不改变持仓。

### 12.3 WATCH_SMALL 买入观察仓

输入：

```text
decision_result.final_action = WATCH_SMALL

quote.price = 5.55
```

执行：

```powershell
python -m ai_trader.paper_trading apply <decision_path> --account paper_default --confirm
```

预期：

- 买入数量向下取整到 100 股整数倍。
- 现金减少。
- `total_quantity` 增加。
- `locked_quantity` 增加。
- `available_quantity` 不增加。
- 生成 `paper_order`、`paper_trade`、`cash_ledger`、`position_ledger`。

### 12.4 当日买入后不能卖出

输入：

```text
同一股票刚买入，available_quantity = 0，locked_quantity > 0
decision_result.final_action = CLEAR
```

预期：

- 不生成成交。
- 订单拒绝或信号阻塞。
- 原因为 `NO_AVAILABLE_QUANTITY` 或 `T_PLUS_ONE_BLOCKED`。

### 12.5 T+1 解锁后卖出

执行：

```powershell
python -m ai_trader.paper_trading rollover --account paper_default --trade-date 2026-07-06
```

再输入：

```text
decision_result.final_action = REDUCE_HALF
```

预期：

- `locked_quantity = 0`。
- `available_quantity` 增加。
- 只解锁已到 `unlock_trade_date` 的锁定明细。
- 卖出数量不超过可卖数量。
- 现金增加。
- 持仓数量减少。

### 12.6 生成每日快照

执行：

```powershell
python -m ai_trader.paper_trading snapshot --account paper_default
```

预期：

- 写入 `account_snapshots.jsonl`。
- 对每只当前持仓写入一条 `position_snapshots.jsonl`。
- 能计算总资产、权益仓位、现金比例、累计收益率。
- 即使当天没有交易，只要仍有持仓，也必须写入当日持仓快照。

## 14. 第二轮完成标准

第二轮代码完成后，应满足：

- 能初始化一个模拟账户。
- 能读取第一轮 `decision_result`。
- `PRE_EVALUATION` 只记录信号，不下单。
- `WATCH_SMALL` 在 `--confirm` 后能生成模拟买入。
- 非交易日或 `NON_TRADING` 决策不能生成模拟成交。
- `paper_signal` 能保存原始决策路径、哈希、策略版本和关键决策字段。
- 成交价格包含滑点时，现金流水不能重复加减滑点。
- 买入数量满足 100 股整数倍。
- 现金不足时订单拒绝。
- 买入后数量进入 `locked_quantity`。
- 买入后写入 `position_locks`，记录买入日和解锁交易日。
- 当日买入不能当日卖出。
- 只能解锁已到 `unlock_trade_date` 的锁定明细。
- 手动或盘前 T+1 解锁后才能卖出。
- `REDUCE_HALF` 和 `CLEAR` 不会卖超过可卖数量。
- 非清仓卖出必须按 100 股整数倍归一化，清仓允许带不足 100 股尾数。
- 每笔成交都关联 `decision_id` 和 `snapshot_id`。
- 每笔成交都写入资金流水和持仓流水。
- 能生成账户快照和持仓快照。
- 每个交易日都能保存当日全部持仓快照，而不是只保存发生交易的股票。
- 能计算总资产、现金比例、权益仓位和累计收益率。
- 模拟盘和实盘概念不混用。
