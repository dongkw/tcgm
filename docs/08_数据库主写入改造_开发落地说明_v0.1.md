# 08 数据库主写入改造开发落地说明 v0.1

> 本文档用于指导“从 JSON/JSONL 文件账本，逐步迁移到 SQLite 主账本”的下一阶段开发。  
> 这一步会改变数据事实源，必须先双写、对账、验证，再考虑让 SQLite 成为主写入。

---

## 1. 当前状态

当前已经完成：

```text
JSON/JSONL 业务写入
  ↓
database.py import-json
  ↓
SQLite 查询索引
  ↓
Web Dashboard 展示
```

也就是说：

- JSON/JSONL 仍然是事实源。
- SQLite 主要是导入后的查询索引。
- Web Dashboard 读 SQLite。
- 业务模块不直接写 SQLite。

当前问题：

- 每次运行新命令后，需要手动 `database.py import-json` 才能刷新界面。
- JSON 和 SQLite 之间可能有延迟。
- 后续界面和插件如果只读 SQLite，会看到旧数据。
- 长期复盘需要更稳定的数据库写入和对账机制。

---

## 2. 改造目标

目标不是一次性删掉 JSON 文件。

目标是分阶段做到：

```text
阶段一：文件写入成功后，同步写 SQLite
阶段二：关键账本双写并自动对账
阶段三：SQLite 作为主写入，JSON 变成导出/归档
```

第一轮改造只做：

```text
双写
对账
失败可降级
不破坏现有 CLI
```

---

## 3. 非目标

本阶段不做：

- PostgreSQL。
- Redis。
- 消息队列。
- 多用户权限。
- 自动实盘下单。
- 删除现有 JSON/JSONL 账本。
- 大规模重写业务模块。

---

## 4. 核心原则

### 4.1 先镜像，后主写

第一阶段 SQLite 是镜像写入。

写入顺序：

```text
业务逻辑生成记录
  ↓
写 JSON/JSONL
  ↓
写 SQLite
  ↓
记录写入结果
```

如果 SQLite 写失败：

- 第一阶段不阻断原有 JSON 账本。
- 必须输出 warning。
- 后续可通过 `database.py import-json` 修复。

等双写稳定后，才调整为：

```text
业务逻辑生成记录
  ↓
写 SQLite
  ↓
导出 JSON/JSONL 归档
```

### 4.2 不允许无记录修账

涉及以下数据，必须可审计：

- 账户现金。
- 持仓数量。
- 可卖数量。
- T+1 锁定数量。
- 成本价。
- 成交记录。
- 资金流水。

### 4.3 不在业务模块里散落 SQL

业务模块不直接拼 SQL。

统一通过：

```text
src/ai_trader/db/writers.py
src/ai_trader/db/readers.py
src/ai_trader/db/stores.py
```

或等价 repository 层写入。

### 4.4 可关闭

双写必须可以通过参数关闭。

建议默认：

```text
--db-write auto
```

含义：

- 如果数据库存在，尝试双写。
- 如果数据库不存在，继续只写 JSON。
- 如果 SQLite 写失败，输出 warning。

后续再支持：

```text
--db-write off
--db-write required
```

## 5. 推荐目录

新增：

```text
src/ai_trader/db/
  writers.py
  readers.py
  sync.py
  consistency.py

src/ai_trader/storage.py
```

职责：

- `writers.py`：把业务记录写入 SQLite 表。
- `readers.py`：后续给业务模块读数据库用。
- `sync.py`：封装“文件写成功后同步写数据库”。
- `consistency.py`：双写对账逻辑。
- `storage.py`：业务模块调用的统一写入门面，避免到处 import db。

第一版可以先不做复杂抽象，但要保留这个方向。

---

## 6. 改造顺序

### 6.1 第一步：数据库写入器

实现：

```text
src/ai_trader/db/writers.py
```

先覆盖低风险表：

```text
strategy_snapshots
decision_results
reports
paper_signals
paper_orders
paper_trades
risk_checks
allocation_plans
order_intents
workflow_runs
intraday_scans
trigger_events
strategy_tuning_records
```

暂不第一批直接写：

```text
accounts
positions
position_locks
cash_ledger
position_ledger
closed_positions
account_snapshots
position_snapshots
```

原因：

- 这些是资金和持仓账本。
- 需要更严格事务和对账。
- 先用导入器保持一致更安全。

### 6.2 第二步：统一文件写入后同步

不要在每个业务模块里手写：

```python
append_jsonl(...)
db_upsert(...)
```

建议新增门面：

```python
append_jsonl_and_mirror(output_root, path, record, mirror_type)
write_json_and_mirror(output_root, path, data, mirror_type)
```

第一批接入：

- `build_decision.py`：策略快照、决策结果。
- `paper_trading.py`：signals/orders/trades。
- `portfolio_plan.py`：risk_checks/allocation_plans/order_intents。
- `workflow.py`：workflow_runs、pre_market_plan、trigger_price_list。
- `intraday_trigger.py`：intraday_scans、trigger_events。
- `strategy_iteration.py`：strategy_tuning_records。

### 6.3 第三步：写入结果记录

每次双写失败必须有记录。

建议新增表或先复用日志：

```text
db_sync_errors
```

字段建议：

```text
sync_error_id
source_path
record_type
record_id
target_table
error_message
payload_json
created_at
```

如果暂不加表，至少写入：

```text
data/logs/db_sync_errors.jsonl
```

后续再导入数据库。

### 6.4 第四步：对账命令

扩展：

```powershell
python .\database.py reconcile
```

当前第一版已落地对账：

- 当前还存在的 `strategy_snapshots/*.json` 是否能在 `strategy_snapshots` 表找到。
- 当前还存在的 `decision_results/*.json` 是否能在 `decision_results` 表找到。
- 当前还存在的 `reports/*.md` 和历史回放报告是否能在 `reports` 表找到。
- 数据库允许保留比文件更多的历史记录，因为文件侧有 `--keep-outputs` 清理策略。

后续再扩展：

- signal/order/trade 引用是否完整。
- risk/allocation/intent 引用是否完整。
- workflow_run 数量是否一致。
- trigger_event 去重是否一致。
- strategy_tuning_record 数量是否一致。

输出：

```text
OK
WARNING
ERROR
```

### 6.5 第五步：资金持仓账本双写

等低风险表稳定后，再处理：

```text
accounts
account_states
positions
position_locks
cash_ledger
position_ledger
closed_positions
account_snapshots
position_snapshots
```

这一步需要事务边界。

例如模拟买入应作为一个逻辑事务：

```text
paper_signal
paper_order
paper_trade
account_state
position
position_lock
cash_ledger
position_ledger
```

第一版 SQLite 可以先用一个连接事务包住数据库写入。

如果数据库写失败：

- 文件账本仍保留。
- 必须要求 `database.py import-json` 修复。
- Web Dashboard 显示“数据库可能落后于文件账本”。

### 6.6 第六步：SQLite 主写入

只有满足以下条件才切：

- 双写运行稳定。
- `reconcile` 连续通过。
- Web Dashboard 主要页面均可直接从 SQLite 查到最新结果。
- 资金、持仓、锁定数量校验没有 ERROR。
- 模拟盘回放能从 SQLite 重建关键状态。

切换后：

```text
SQLite 主写入
JSON/JSONL 作为归档导出
```

## 7. 第一批具体改造范围

建议下一轮代码只做这几个点：

### 7.1 新增 writers

实现低风险写入函数：

```python
write_strategy_snapshot(conn, snapshot, artifact_id=None)
write_decision_result(conn, decision, artifact_id=None)
write_paper_signal(conn, signal)
write_paper_order(conn, order)
write_paper_trade(conn, trade)
write_risk_check(conn, risk_check)
write_allocation_plan(conn, allocation_plan)
write_order_intent(conn, order_intent)
write_workflow_run(conn, workflow_run)
write_intraday_scan(conn, scan)
write_trigger_event(conn, event)
write_strategy_tuning_record(conn, record)
```

这些函数可以复用 `importers.py` 里的字段映射逻辑，但不能直接依赖“读取文件再导入”的流程。

### 7.2 新增 sync 门面

实现：

```python
mirror_record(output_root, record_type, record)
```

行为：

- 自动定位 `data/ai_trader.sqlite3`。
- 数据库不存在时跳过。
- 数据库存在时 upsert。
- 失败时记录 warning。

### 7.3 接入 build_decision

第一批最适合先接：

```text
strategy_snapshots
decision_results
reports 索引
```

原因：

- append/写入风险低。
- 不涉及资金。
- 失败后可由 `import-json` 修复。
- Web Dashboard 马上能看到新决策。

### 7.4 接入 strategy_iteration

第二个低风险接入点：

```text
strategy_tuning_records
reports 索引
```

原因：

- 也是追加型记录。
- 不影响交易账本。
- 对策略闭环很有用。

## 8. 不建议第一批改的地方

不建议第一批改：

```text
portfolio.py
paper_trading.py apply_buy_trade/apply_sell_trade
historical_replay.py portfolio 写入
```

原因：

- 涉及现金和持仓状态。
- 一次操作会改多张表。
- 当前 JSON 写入已经能跑通。
- 应先有 reconcile 再改。

## 9. 对账规则

### 9.1 记录数对账

示例：

```text
data/decision_results/*.json count == decision_results count
data/strategy_snapshots/*.json count == strategy_snapshots count
data/paper_trading/signals.jsonl count == paper_signals count
```

注意：

- replay 子目录需要单独按 replay_id 对账。
- 报告可能有历史保留策略，不能简单按文件数一刀切。

### 9.2 关键 ID 对账

需要检查：

```text
decision_id
snapshot_id
signal_id
order_id
trade_id
risk_check_id
allocation_id
intent_id
workflow_run_id
scan_id
trigger_event_id
iteration_id
```

### 9.3 资金持仓对账

后续改持仓双写前，必须已有：

```text
account_state.total_assets = available_cash + frozen_cash + market_value
position.market_value = total_quantity * market_price
sum(position.market_value) = account_state.market_value
sum(open locks.remaining_locked_quantity) = position.locked_quantity
```

## 10. Web Dashboard 配合

已完成的状态提示：

```text
数据库最新导入时间
reconcile 状态
数据库是否落后于文件账本
```

当前实现：

- 顶部状态栏显示 `同步 OK / ERROR / MISSING`。
- 数据健康页显示 `reconcile` 状态、文件数量、数据库行数、同步问题列表。
- 数据健康页提供“执行对账”按钮。
- JSON API 提供 `GET /api/sync-status`。

如果数据库落后，界面不能假装正常。

## 12. 验收标准

第一批双写改造完成后必须满足：

- 没有 SQLite 时，原有 CLI 仍能正常跑。
- 有 SQLite 时，新生成的决策能自动出现在 Web Dashboard。
- SQLite 写失败不会损坏 JSON 文件。
- `database.py import-json` 仍可作为修复手段。
- `database.py validate` 仍可通过。
- 新增低风险表双写不引入重复记录。
- 重复执行同一导入不会造成业务表膨胀。

资金持仓双写完成前，不能宣布 SQLite 是最终主账本。
