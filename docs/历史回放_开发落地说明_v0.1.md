# 历史回放开发落地说明 v0.1

> 本文档用于指导下一阶段 `historical_replay` 的开发。  
> 当前只写设计，不写代码。  
> 历史回放的目标是快速验证策略是否明显有效或明显无效，不是第一版就做成专业机构级回测系统。

---

## 1. 核心定位

历史回放不是重新写一套交易系统。

它的定义是：

```text
历史回放 = 模拟盘的历史时间版本
```

区别只在时间来源：

```text
模拟盘：
  时间来自真实今天，每天往后运行

历史回放：
  时间来自用户指定历史区间，按交易日逐日推进
```

历史回放必须复用现有模拟盘和组合链路：

```text
strategy_snapshot
  ↓
decision_result
  ↓
risk_check
  ↓
allocation_plan / order_intent
  ↓
paper_order / paper_trade
  ↓
account / position / ledger / snapshot
  ↓
performance_report
```

禁止为了回测结果好看，单独写一套不遵守 T+1、手续费、滑点、仓位限制的简化交易逻辑。

---

## 2. 第一版要解决的问题

第一版历史回放要回答：

1. 这套策略在过去一段时间是否明显不行。
2. 买点、卖点、止损、仓位规则有没有明显问题。
3. 策略修改前后，历史决策有什么差异。
4. 模拟盘规则在历史时间里能不能连续跑通。
5. 哪些失败样本应该沉淀为后续测试样本。

第一版不承诺：

- 证明策略一定赚钱。
- 做全市场专业回测。
- 做分钟级或 tick 级撮合。
- 精确处理所有财报和公告点时数据。
- 自动优化参数。
- 接入实盘。

---

## 3. 重要结论：当前 JSON 不够直接回放

当前已有：

```text
data/stock_json/stock_data_<code>.json
```

它适合做当前时点的单票决策，但不够做严谨历史回放。

原因：

- 它通常只保存当前最新快照。
- 财报、公告、估值、技术指标可能是当前口径。
- 如果直接把当前财报放进过去某天，会产生未来函数。
- 如果用当前均线或当前价格回放过去，也会污染结果。

所以历史回放必须新增历史数据层。

第一版至少需要：

```text
历史日线行情
历史复权价格
历史技术指标
交易日序列
每个历史交易日可见的数据快照
```

财报和公告如果没有准确披露时间，第一版不能作为硬规则输入。

---

## 4. 两种回放模式

### 4.1 REPLAY_LITE

第一版建议先做这个。

用途：

- 快速验证交易链路。
- 快速验证价格、趋势、均线、止损、仓位规则。
- 快速发现明显无效策略。

数据范围：

```text
历史日线 K 线
前复权或后复权价格
成交量
涨跌幅
MA20 / MA60
20 日高低点
ATR 等技术指标
```

限制：

- 不使用没有点时生效日期的财报字段做硬规则。
- 不使用没有披露时间的公告字段做硬规则。
- 财务和公告相关规则只能输出 `UNKNOWN`、`DATA_NOT_AVAILABLE` 或降级。
- 结果只能说明价格/技术/执行规则表现，不能证明完整基本面策略有效。

### 4.2 PIT_STRICT

后续严谨版。

PIT 表示 point-in-time，即历史当时视角。

必须支持：

- 财报按公告披露日生效。
- 公告按披露时间生效。
- 停牌、复牌、涨跌停。
- 历史成分变化。
- 数据源修正和版本。
- 当日盘后数据不能影响当日盘中决策。

第一版不直接做 PIT_STRICT，但所有字段设计要为它预留。

## 5. 第一版输入

建议命令形态：

```powershell
python .\historical_replay.py run --account replay_default --symbols 002563 --start 2025-01-01 --end 2025-12-31 --cash 100000 --mode lite
```

核心输入：

```text
account
symbols
start_date
end_date
initial_cash
replay_mode
strategy_version
execution_mode
cost_model
data_dir
output_dir
benchmark
```

字段说明：

| 字段 | 说明 |
|---|---|
| `account` | 回放账户 ID，必须和真实模拟盘账户隔离 |
| `symbols` | 股票代码列表，第一版可以先单票 |
| `start_date` | 回放开始日期 |
| `end_date` | 回放结束日期 |
| `initial_cash` | 初始虚拟本金 |
| `replay_mode` | `REPLAY_LITE` 或后续 `PIT_STRICT` |
| `strategy_version` | 策略版本 |
| `execution_mode` | 默认 `NEXT_OPEN` |
| `cost_model` | 手续费、印花税、滑点参数 |
| `benchmark` | 可选，对比指数或标的本身买入持有 |

## 6. 历史数据结构

### 6.1 historical_daily_bars

建议第一版文件：

```text
data/historical/daily_bars/<symbol>.jsonl
```

每行一条交易日行情：

```json
{
  "symbol": "002563",
  "trade_date": "2025-01-02",
  "open": 5.12,
  "high": 5.30,
  "low": 5.05,
  "close": 5.22,
  "pre_close": 5.10,
  "volume": 12345600,
  "amount": 65000000.0,
  "pct_change": 2.35,
  "adj_factor": 1.0,
  "price_adjustment": "qfq",
  "is_suspended": false,
  "is_limit_up": false,
  "is_limit_down": false,
  "source": "akshare_or_other",
  "fetched_at": "2026-07-04T20:00:00+08:00"
}
```

要求：

- 一个 `symbol + trade_date` 只能有一条主记录。
- 价格复权口径必须固定，不能一会儿前复权一会儿不复权。
- `pre_close` 必须保存，用于判断涨跌停。
- 停牌日必须能识别。
- 数据源和抓取时间必须记录。

### 6.2 historical_features

建议第一版文件：

```text
data/historical/features/<symbol>.jsonl
```

每行保存当日收盘后可计算出的指标：

```json
{
  "symbol": "002563",
  "trade_date": "2025-01-02",
  "ma20": 5.01,
  "ma60": 4.82,
  "high_20d": 5.55,
  "low_20d": 4.66,
  "change_20d_pct": 8.7,
  "change_60d_pct": 15.2,
  "atr14_pct": 3.1,
  "source_bar_end_date": "2025-01-02",
  "created_at": "2026-07-04T20:00:00+08:00"
}
```

注意：

- `2025-01-02` 的特征只能用于 `2025-01-02` 盘后或 `2025-01-03` 盘前。
- 如果要在 `2025-01-02` 开盘前决策，只能使用上一交易日的特征。

### 6.3 historical_effective_events

后续 PIT_STRICT 使用。

建议文件：

```text
data/historical/effective_events/<symbol>.jsonl
```

字段：

```text
symbol
event_type
event_time
effective_from
effective_to
source
payload
```

用途：

- 财报。
- 公告。
- ST。
- 停牌。
- 分红除权。
- 限售解禁。
- 减持公告。

第一版可以先不实现，但不能用当前最新事件回填历史。

## 7. 每日可见快照

历史回放不能直接把整段历史数据交给策略。

每天必须构建“当日可见”的 `strategy_snapshot`。

盘前回放口径：

```text
目标交易日 = D
可见行情截止 = D 的上一交易日收盘
source_quote_trade_date = previous_trade_date(D)
decision_time = D 08:45:00 Asia/Shanghai
session_name = PRE_MARKET
```

盘后回放口径：

```text
目标交易日 = D
可见行情截止 = D 当日收盘
source_quote_trade_date = D
decision_time = D 15:30:00 Asia/Shanghai
session_name = POST_MARKET
```

第一版建议采用盘前口径：

```text
用 D-1 收盘数据做 D 盘前计划
在 D 开盘按规则模拟成交
D 收盘后生成账户快照
```

这样更符合当前已经实现的盘前工作流。

## 8. 每日回放流程

第一版日频流程：

```text
init_replay_account

for trade_date in replay_dates:

```

```text

日频
盘前决策
开盘成交
收盘估值
```

## 9. 成交规则

默认执行模式：

```text
execution_mode = NEXT_OPEN
```

在盘前口径下，含义是：

```text
用 D-1 收盘后已知数据生成 D 盘前信号
如果通过风控和组合分配
在 D 开盘价附近模拟成交
```

买入成交价：

```text
fill_price = open_price * (1 + slippage_rate)
```

卖出成交价：

```text
fill_price = open_price * (1 - slippage_rate)
```

保守成交限制：

```text
is_suspended = true        -> 不成交
买入且 is_limit_up = true   -> 不成交
卖出且 is_limit_down = true -> 不成交
open_price 缺失             -> 不成交
```

拒绝原因：

```text
SUSPENDED
LIMIT_UP_BUY_BLOCKED
LIMIT_DOWN_SELL_BLOCKED
OPEN_PRICE_MISSING
PRICE_TIME_MISMATCH
```

交易成本继续复用现有 `cost_model`：

```text
commission_rate
min_commission
stamp_tax_rate_sell
slippage_rate
```

## 10. A 股规则必须继承

历史回放必须遵守：

- T+1。
- 100 股整数买入。
- 清仓可以卖不足 100 股尾数。
- 非清仓卖出按 100 股整数倍。
- 买入不能超过可用现金。
- 卖出不能超过可卖数量。
- 现金保留线。
- 单票仓位上限。
- 单日买入额度。
- 涨停买不到。
- 跌停卖不出。
- 停牌不成交。

这些规则应复用现有：

```text
portfolio.py
paper_trading.py
cost_model.py
risk_control.py
portfolio_construction.py
```

如果为了回放方便绕过这些规则，回放结果无效。


解释：

- 总收益不够，必须看最大回撤。
- 胜率不够，必须看盈亏比。

- 回放交易太少时，不允许给强结论。

报告里必须标注：

```text

```

## 15. 错误归因

`error_cases.jsonl` 记录需要复盘的样本。

错误类型：

```text
STRATEGY_ERROR
DATA_ERROR
TIME_ERROR
EXECUTION_ERROR
RISK_CONTROL
MARKET_NOISE
OVERFITTING_RISK
SAMPLE_TOO_SMALL
```

触发场景：

- 单笔亏损超过阈值。
- 连续亏损。
- 最大回撤创新高。
- 买入后很快触发止损。
- 因数据缺失导致大量 `DATA_BLOCKED`。
- 因涨跌停导致无法成交。
- 策略修改后结果大幅变化。

每条错误样本至少记录：

```text
symbol
trade_date
decision_id
trade_id
entry_reason
exit_reason
loss_pct
suspected_cause
source_snapshot_path
source_decision_path
```

## 16. 禁止未来函数规则

历史回放最重要的是时间正确。

禁止：

- 用 `D` 日收盘价生成 `D` 日开盘前买入信号。
- 用当前最新财报填充过去日期。
- 用当前公告列表判断过去风险。
- 用未来最高价、最低价决定历史买卖。
- 用 `end_date` 后的数据计算回放区间内指标。
- 回放中途根据最终收益动态改参数。
- 回放失败后静默跳过亏损交易。

必须保证：

- `strategy_snapshot.source_quote_trade_date < trade_date`，如果是盘前口径。
- `decision_time` 固定为历史当日时间。
- 技术指标只用数据截止日前的数据计算。
- 财报和公告必须有 `effective_from` 才能进入硬规则。
- 数据缺失必须显式记录，不允许补造。

## 19. 第一版开发顺序

建议按下面顺序开发：

1. 定义 `historical_daily_bars` JSONL 格式。
2. 写历史日线读取和校验。
3. 从日线生成 `historical_features`。
4. 实现历史交易日序列。
5. 实现盘前口径的 `replay_time_context`。
6. 实现当日可见 `stock_json` 构建。
7. 复用第一轮生成 `strategy_snapshot` 和 `decision_result`。
8. 初始化独立回放账户目录。
9. 每个交易日执行 `rollover`。
10. 运行风控和组合构建。
11. 用开盘价模拟成交。
12. 收盘后按收盘价估值并生成快照。
13. 写 `daily_replay_records.jsonl`。
14. 计算绩效指标。
15. 生成 `replay_report.md`。
16. 用一只股票半年数据做验收。

## 20. 第一版验收标准

代码完成后，应满足：

- 能读取一只股票半年或一年历史日线。
- 能按交易日逐日推进。
- 能保证盘前决策只看到上一交易日收盘数据。
- 能生成每日 `strategy_snapshot` 和 `decision_result`。
- 能复用现有模拟盘 T+1 和成本模型。
- 买入遵守 100 股整数倍。
- 当日买入不能当日卖出。
- 涨停买入和跌停卖出能被保守拦截。
- 能生成每日账户快照。
- 能输出收益率、最大回撤、胜率、盈亏比。
- 能记录数据缺失和阻塞原因。
- 回放产物不污染真实模拟盘目录。
- 重复运行会生成新的 `replay_id`，不会覆盖旧回放结果。

## 21. 第一版不做的事

明确不做：

- 分钟级撮合。
- Tick 级回放。
- 自动参数优化。
- 多策略组合。
- 严格财报点时数据库。
- 全市场批量回测。
- 图形化收益曲线。
- AI 自动修改策略。

这些能力必须等第一版日频历史回放跑通后再加。

---

## 22. 开发前必须确认的问题

进入编码前，至少确认：

1. 历史日线数据来源用哪个。
2. 价格复权口径用前复权、后复权还是不复权。
3. 第一版先单票还是支持多票。
4. 第一版策略是否允许 `REPLAY_LITE` 禁用财务硬规则。
5. 默认成交是开盘价还是下一交易日开盘价。
6. 基准用沪深 300、中证 500，还是个股买入持有。

建议默认答案：

```text
历史日线：先用当前数据源能稳定拿到的日线
复权口径：前复权
标的范围：先单票，再多票
回放模式：REPLAY_LITE
成交模式：D 盘前信号，D 开盘成交
基准：个股买入持有，后续加指数
```

---

## 23. 策略迭代闭环

历史回放不是为了生成一份好看的收益报告。

它真正要服务的是后续策略升级：

```text
策略规则
  -> 历史回放
  -> 绩效和错误样本
  -> 策略复盘
  -> 修改买入 / 卖出 / 风控 / 仓位规则
  -> 再次回放
  -> 模拟盘观察
```

每次回放后都应该沉淀：

- 本次策略版本。
- 修改了哪些规则。
- 回放区间和标的范围。
- 总收益、胜率、盈亏比、最大回撤、交易次数、平均持仓天数。
- 最大亏损样本。
- 明显错误买入。
- 明显错误卖出。
- 漏掉但事后看应该买入的样本。
- 下一次策略修改方向。

这一步是系统后续最重要的工作。

`v0.1` 的意义是把流程跑通；真正让系统变有用的是不断用历史回放和模拟盘结果反推策略缺陷。

---

## 24. 方向边界

历史回放服务于赚钱目标，但不能为了结果好看牺牲真实性。

必须坚持：

```text
宁可结果难看
也不要未来函数

宁可数据缺失
也不要编造数据

宁可先跑单票
也不要一开始做全市场假精确

宁可暴露策略无效
也不要调参到历史最优
```

历史回放第一版真正有价值的结果，不一定是“策略赚钱”，而是：

- 找出哪些规则明显无效。
- 找出哪些数据必须补。
- 找出哪些交易限制必须加入。
- 找出哪些亏损样本需要复盘。
- 让后续策略修改有验证依据。
